# Skill: Webhook Guard & Recursion Safety

Эта инструкция обязательна для всех агентов, работающих с Telegram API в среде Edge Functions.

## 1. Понимание риска
В Edge Functions (Supabase) входящий вебхук от Telegram запускает выполнение функции. Если внутри функции бот отправляет сообщение, которое Telegram считает "новым событием" (например, редактирование собственного сообщения без проверки), это может создать бесконечную петлю вызовов.

## 2. Золотые правила
- **Проверка ID сообщения**: Всегда проверяй, не является ли `message_id` или `from.id` техническим или уже обработанным.
- **Минимальный Payload**: Не передавай в `sendMessage` данные, которые могут быть интерпретированы как команды, если ты просто логируешь ошибку.
- **Return Statement**: В Deno/Supabase ОБЯЗАТЕЛЬНО возвращай `new Response("OK", { status: 200 })` максимально быстро после успешной обработки, чтобы избежать повторных попыток отправки вебхука со стороны Telegram (таймаут).

## 3. Обработка ошибок
Никогда не отправляй текст ошибки в чат в блоке `catch`, если ошибка произошла при самой отправке сообщения. Это создаст цикл: Ошибка отправки -> Catch -> Попытка отправить ошибку -> Ошибка отправки...

## 4. Чек-лист перед коммитом
- [ ] Добавлен ли `return` после каждой логической ветки?
- [ ] Есть ли в коде защита от обработки собственных сообщений бота?
- [ ] Оптимизирован ли размер ответа вебхуку?
